Project Name: AroundCornersWorldApp

Intention: This project objective is to demonstrate the benefits and easy of use of the Microsoft Azure App Services (PaaS Platform), as well as how to use it in combination with Azure Resource Manager Templates to the environment automation (creation, update and etc), and integration with Github (for Continuous Deployment).

Motivation: To help users/developers that want to to learn how create a web application composed of many services and different programming languages, all linked/backed by github repositories for continuous deployment, and without the burden of environment administration (e.g.: Operating System install/upgrade, web server setup, and etc).

Aplicability: This assets can be used by Developers, DevOps and Application Architects.

Structure:

1 - A Database (Azure PaaS - NoSQL Database);

AroundCornersDB - DocumentDB;

2 - Two Microservices (Azure PaaS - App Service/API App):

AroundCornersCityLocatorAPI - Implemented in NodeJS;

AroundCornersTimezoneAPI - Implemented in PHP;

3 - A WEB Frontend (Azure PaaS - App Service/WEB App):

AroundCornersWorldApp - Implemented in PHP;

Introduction

This is a personal and independent project created solely for the purpose of Cloud education and sharing of architectural designs for cloud application deployments, recommended good practices, and showcase:

1) Important concepts around modern application structure and deployments on the Cloud;

2) How to leverage automation features (e.g.: flexibility and predictability) that declarative languages such as Azure JSON Templates (ARM) provides;

3) The benefits of Platform as a Service (PaaS) over Infrastructure as a Service (IaaS), from the point of view of: easy-of-use, focus on your code, no handling of Operating System patches/upgrades, bootstrap/configuration of the environment (e.g.: NodeJS installation, configuration and/or WEB Server setup), and etc;

4) How developers can use the power of Azure PaaS Cloud Services to just focus on their code, as all (continuous) deployments and production code can be backed/linked to GIT repositories. All you need is a Microsoft Azure Subscription, a JSON template, and your code (that can be on your local repository, Azure built-in or on Github);

5) How to use NoSQL Databases and show different use cases for this technology, opposed to just go with SQL DB's in scenarios where they are not needed or even should be avoided.

In the end, all you need to know is your favorite programming language (e.g.: NodeJS) and JSON (what I think every programmer eats for the breakfast)!

After you have deployed the whole application (what you do just with JSON files), you will be able to just work on your code (e.g.: on github.com), and everytime you push to your master branch (or any other branch you choose to it), your code will be automatically deployed on Azure!

The AroundCorners' SOURCE CODE:

Here you have the actual repositories (source code) of each service above. You just need to fork/clone it, and you are ready to go!

Database

Code Repository | JSON ARM Template

This is the Database for one of our microservices (AroundCornersCityLocatorAPI). It is a "static" DB that is loaded from a dump downloaded from GeoNames.org. As I believe cities do not change locations, after the first load there is no procedure to update it. But for sure an Azure WebJob or similar service could be used to load a new dump every day or so... one of the use cases that I can think of are new cities (the dump used here is for cities around a certain population). The database is normalized to lowercase.

AroundCornersCityLocatorAPI

Code Repository | JSON ARM Template

This NodeJS application provides the coordinates (Latitude and Longitude) of a city on globe. As one of the services of our World Application is to point a arbitrary city on the earth's map, we use this API to tell us the coordinates of any given city the user requests.

AroundCornersTimezoneAPI

Code Repository | JSON ARM Template

Another service of our World Application is called "Time". What this service does is to inform the full date and time (e.g.: 15:56 on Tuesday 17th November 2015 at America/Sao_Paulo), as it receives a request containing an timezone region (e.g.: America/Sao_Paulo). This very simple code is written in PHP...

AroundCornersWorldApp

Code Repository | JSON ARM Template

This is the FrontEnd WEB Application, and the "face" of the AroundCorners World App. The main page is a simple HTML WEB page that use public and app specific assets files (e.g.: CSS and Image files), that are hosted on public sites or Azure Blob Storage. This is the page that references the two PHP pages that use the two above API's (map.php and time.php). One of the important learning points here is that two or more (e.g.: dozens microservices/API's), are the recommended way to build a scalable and resilient web application. The idea here is that more API services can be added to the AroundCorners World App, as the architecture provides the general structure to it. Look that the API's used on the application use different languages (NodeJS and PHP), and that does not impact the overall solution, as the communication between them is always JSON. Many popular web applications (e.g.: e-commerce sites) are build with this principle, and as you build strong and popular web services/api's for your own utilization, you can use Azure API Management for example, to expose these API's to be consumed (and paid!) by customers.

Other minor concepts and recommendations are implemented in this example app, as well as others that should be integrated over time (e.g.: Add Azure CDN for assets and Azure API Management to profit from our API's). This should be on ongoing and exciting project, where I plan to add and integrate Azure Cloud features and services to help new or experienced IT Pro's, to help the understanding and how to leverage the featureful Microsoft Azure Cloud services. As Cloud services are evolving really fast, the idea is that anyone can find in this project, at least one feature or point of view that can be of benefit. Just hope that ends up being true...

IMPORTANT: The main point of this project IS NOT focused on the "coding" per se, or any specific programming techniques or algorithm. All the coding and algorithms used here have the main purpose to create working parts to demonstrate an architectural concept and/or to have minimal application functionalities. So, users/readers/developers and etc can correlate this example solution presented here with their own projects and use cases. The code here in a specific API can be deliberately wrong, for example, to show how an resilient application should handle such scenario; or to have an infinite loop or literal sleeping code to discuss Micro-services SLA's and application behavior. Well, other coding can just be plain wrong by the simple reason that I'm a really bad programmer and cannot see any difference between O(n) and O(log n).

USE AT YOUR OWN RISK

This project was created to help users to get used to cloud concepts, architectural patterns and recommended practices. Both App API Applications on this example project are publicly accessible as the purpose is to easy showcase the API's response, and etc. For production use you may should take security into consideration, and probably add some kind of access control to your API's (e.g.: Token). I plan to add some example of this when I do integrate the Azure API Management into this project.
